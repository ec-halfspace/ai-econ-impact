# Scoring strategy

During the eval, the `model x harness`generates files (the deliverables) - pdfs, excel sheets etc. based on the requirements in the task prompt - that Inspect stores in the sample’s `.eval` output.

Then, the pairwise_judge scorer loads task metadata, pulls the model’s deliverables from the log, loads the golden deliverables (if they exist), and runs the judge that compares the two side by side.

During a scoring round by the judge, the following steps occur:

1. The `pairwise_judge` looks up at the task by task_id in `data/gdpval_tasks.parquet` (previously downloaded from HF GDPval dataset).
2. From that row, it reads the task prompt and the rubric text (`rubric_pretty` is the human readable version that the judge also reads; `rubric_json` is only in the .parquet for metadata). In the original GDPval paper, task writers create **detailed scoring rubrics** for consistency and transparency, but their Autograder didn’t use them. It resolves which golden files belong to the task using the `deliverable_files` column in the `.parquet`, then loads the actual files from `data/golden_deliverables/{task_index}/` . See an example of a rubric below; note that it is different for every task; the ‘scores’ shown there are guides only; the judge DOES NOT weight them in when considering the final score.

![Screenshot 2026-06-17 at 16.54.49.png](Scoring%20strategy/Screenshot_2026-06-17_at_16.54.49.png)

1. This rubric text is pasted into the judge’s user prompt under a “`RUBRIC (what a good solution should achieve)`” heading, together with the task prompt and the sandbox paths to solutions A and B. The judge is told to check completeness, correctness, professional quality, and adherence to the rubric. This can be seen in `gdpval-judge/prompts.py`:

![Screenshot 2026-06-17 at 17.07.29.png](Scoring%20strategy/a9dd9775-bb2a-4f0a-a217-b8af90e955d0.png)

Note that the rubric is used only during the scoring, the eval doesn’t see it, only the task prompt!!!!

You can see this prompt in Inspect view in the Transcript → scroll down to scoring:

![Screenshot 2026-06-17 at 17.35.45.png](Scoring%20strategy/Screenshot_2026-06-17_at_17.35.45.png)

1. Both directories are mounted read-only into a Docker sandbox (`gdpval-arm64`). The judge is a PydanticAI agent with an `execute_python` tool that can open spreadsheets, PDFs, Word docs, and similar files using libraries like `openpyxl` and `pdfplumber`, available in the sandbox:

![Screenshot 2026-06-17 at 17.07.29.png](Scoring%20strategy/75514e7d-b6c5-4ad1-ac50-c7214c358d22.png)

So far, the judge used gpt-5 (medium reasoning), but that can also be modified if needed. 

1. To overcome bias, the model and the golden deliverables are marked randomly as A and B. Then, the judge inspects both solutions, and returns a structured verdict: prefer A, prefer B, or TIE, plus a short reasoning string. These are then mapped onto 1 if the model wins, 0 if the golden wins, 0.5 if it’s a tie. 
2. In the case where the judge timed out (default is 300s) or simply failed, the score will be marked as E (might register as zero in the folder view in Inspect, that’s more like a Nan). If there is no golden deliverable available for comparison, the score will be marked as N (again might register as zero outside the log). An empty model output (could happen e.g. due to runtime error, concurrency issue etc.) will be scored 0.
3. By default, each .eval log is judged 3 times and the output is the average of the scores.
